## Overview

Patient Interaction Practice Tool uses a two-tier dependency management approach:

- **Python services** (Docker Lambda & ECS): `requirements.in` → `pip-compile` → `requirements.txt` for reproducible builds
- **Node.js services** (Lambda layers, socket server): `package.json` → `npm ci` → `package-lock.json` for deterministic installs
- **Lambda layers**: Pre-built ZIP files checked into the repository at `cdk/layers/`

This document covers the full dependency map across all services, the lock file strategy that ensures reproducible builds, and the procedures for adding, updating, and deploying dependency changes.

### Socket Server Hybrid

The socket server (`cdk/socket-server/`) is unique — it runs both Node.js and Python in the same Docker container:

- **Node.js** (`package.json`): Express, Socket.IO, WebRTC (werift), AWS SDK
- **Python** (`requirements.txt`): Nova Sonic streaming, Bedrock SDK, LangChain (for RAG during voice)

The Dockerfile installs both runtimes:

```dockerfile
# Node.js dependencies
COPY package.json package-lock.json ./
RUN npm ci

# Python dependencies
COPY requirements.txt ./
RUN pip3 install --break-system-packages -r requirements.txt
```

When updating socket server dependencies, edit the appropriate file (`package.json` for Node.js, `requirements.txt` for Python) and rebuild.

## Lock File Strategy

### Python: requirements.in to requirements.txt

The `requirements.in` file lists top-level dependencies with loose version constraints. `pip-compile` (from the `uv` or `pip-tools` package) resolves and pins all transitive dependencies into `requirements.txt`.

**File structure:**

```text
requirements.in          ← Human-edited, top-level deps
requirements.txt         ← Machine-generated, fully pinned (DO NOT EDIT)
```

**Locking process:**

```bash
# Using uv (recommended — faster)
uv pip compile requirements.in -o requirements.txt

# Or using pip-tools
pip-compile requirements.in -o requirements.txt
```

### Node.js: package.json to package-lock.json

Node.js dependencies are managed with npm. The `package-lock.json` ensures deterministic installs.

```bash
npm install              # Updates package-lock.json
npm ci                   # Clean install from lock file (used in CI/Docker)
```

### Lambda Layers (Node.js)

Node.js Lambda layers are built by installing dependencies into a `nodejs/` directory structure and zipping:

```bash
mkdir -p nodejs
cd nodejs
npm init -y
npm install <package-name>
cd ..
zip -r <layer-name>.zip nodejs/
```

### Lambda Layers (Python)

Python Lambda layers require building on a Linux/x86_64 platform to match the Lambda runtime. Docker provides the correct build environment:

```bash
docker run --rm -v $(pwd):/out -w /out public.ecr.aws/lambda/python:3.12 \
  pip install <package-name> -t python/lib/python3.12/site-packages/

zip -r <layer-name>.zip python/
```

## Update Procedures

### Adding a New Python Dependency

1. Edit `requirements.in` in the relevant service directory:

   ```text
   # cdk/text_generation/requirements.in
   langchain
   langchain-aws
   my-new-package>=1.0.0    # ← Add here
   ```

2. Recompile the lock file:

   ```bash
   cd cdk/text_generation
   uv pip compile requirements.in -o requirements.txt
   ```

3. Verify the Docker build works:

   ```bash
   docker build -t test-textgen ./cdk/text_generation
   ```

4. Commit both files (`requirements.in` and `requirements.txt`).

### Updating an Existing Python Dependency

1. Update the version constraint in `requirements.in` (or leave it loose to get the latest compatible version).

2. Recompile:

   ```bash
   cd cdk/text_generation
   uv pip compile --upgrade-package <package-name> requirements.in -o requirements.txt
   ```

   Or to upgrade all dependencies:

   ```bash
   uv pip compile --upgrade requirements.in -o requirements.txt
   ```

3. Test the build and run tests.

4. Commit both files.

### Adding a New Node.js Lambda Layer

1. Create the layer directory structure:

   ```bash
   mkdir -p /tmp/layer/nodejs
   cd /tmp/layer/nodejs
   npm init -y
   npm install <package-name>
   cd ..
   zip -r <layer-name>.zip nodejs/
   cp <layer-name>.zip <path-to-repo>/cdk/layers/
   ```

2. Reference the layer in the CDK stack (`cdk/lib/api-service-stack.ts`):

   ```typescript
   const myLayer = new lambda.LayerVersion(this, "my-layer", {
     code: lambda.Code.fromAsset("./layers/my-layer.zip"),
     compatibleRuntimes: [lambda.Runtime.NODEJS_22_X],
     description: "Contains the my-package library",
   });
   ```

3. Commit the ZIP file and CDK changes.

### Rebuilding a Python Lambda Layer

Python layers must be built on a compatible platform (Linux x86_64):

```bash
# Build psycopg2 layer using Docker
docker run --rm -v $(pwd)/cdk/layers:/out -w /out public.ecr.aws/lambda/python:3.12 \
  bash -c "pip install psycopg2-binary -t python/lib/python3.12/site-packages/ && \
           cd /out && zip -r psycopg2.zip python/ && rm -rf python/"
```

### Rebuilding a Node.js Lambda Layer

```bash
# Build node-pg-migrate layer
mkdir -p /tmp/node-pg-migrate/nodejs
cd /tmp/node-pg-migrate/nodejs
npm init -y
npm install node-pg-migrate pg
cd ..
zip -r node-pg-migrate.zip nodejs/
cp node-pg-migrate.zip <path-to-repo>/cdk/layers/
```

### Deploying After Dependency Changes

After modifying dependencies:

- **Docker services** (text_generation, data_ingestion, voice-agent, socket-server): Push a commit to trigger the CI/CD pipeline, or manually build and push to ECR.
- **Lambda layers**: Deploy the CDK stack that references the layer:

  ```bash
  npx cdk deploy <STACK PREFIX HERE>-Api -c StackPrefix=<STACK PREFIX HERE> -c githubRepo=<REPO NAME HERE>
  ```

### Best Practices

1. **Never edit `requirements.txt` directly.** Always edit `requirements.in` and recompile.
2. **Pin critical packages** in `requirements.in` when exact versions matter (e.g., `PyMuPDF==1.24.10`).
3. **Use `npm ci`** (not `npm install`) in CI/CD and Docker builds for deterministic installs.
4. **Test Docker builds locally** before pushing dependency changes.
5. **Keep layer ZIPs small.** Only include the minimum packages needed. Large layers increase cold start times.
6. **Commit lock files.** Both `requirements.txt` and `package-lock.json` must be committed to ensure reproducible builds across environments.
